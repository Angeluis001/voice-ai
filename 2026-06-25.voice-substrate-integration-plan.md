# Voice Substrate Integration Plan

**Date:** 2026-06-25
**Source docs:** `2026-04-30.voice-substrate.md`, `policies-voice-substrate.md`
**Status:** Draft integration plan with an active drachtio-based implementation path

## Goal

Integrate the locked voice substrate into the broader ValOS stack without reintroducing deprecated patterns.

Target outcomes:

- Device-side voice uses Gemma 4 E4B native audio input directly.
- Telephony voice uses the Kamailio + FreeSWITCH wire stack with a dedicated SIP application layer and STT sidecar.
- Voice output stays behind a single `VoiceChannel` abstraction and expands by tier.
- val-switch becomes the orchestration point for routing, threading, policy, and TTS selection.

## Locked decisions to preserve

- Do not add Whisper or any STT sidecar on laptops, kiosks, or Physical AI devices.
- Do use a telephony STT sidecar because call media lives in FreeSWITCH, not on the device.
- Do keep Kokoro as the default local TTS path.
- Do extend the existing voice output abstraction instead of creating ad hoc TTS integrations.
- Do treat Kamailio, FreeSWITCH, and MeshCentral as fixed architectural anchors, while the current implementation replaces Jambonz with drachtio by explicit project decision.
- Do keep cloud STT/TTS behind privacy and cost policy gates.

## Integration tracks

### 1. Device voice path

Objective: establish a direct microphone-to-model round trip on the operator device.

Deliverables:

- Audio capture module that emits 16kHz PCM frames.
- Request adapter for Gemma 4 E4B audio input.
- Structured response contract for `{ transcript, intent, response }`.
- Playback hook that routes generated text into the TTS layer.

Implementation notes:

- Use one inference path for transcript, intent, and reply generation.
- Avoid building separate STT and intent microservices for device voice.
- Normalize request and response schemas early so the same conversation objects can later be threaded into val-switch.

Acceptance criteria:

- Operator can speak into a laptop microphone and receive a local transcript plus spoken reply.
- The path works without any Whisper container or remote STT dependency.

### 2. Voice output layer

Objective: make TTS extensible while keeping Kokoro as the default.

Deliverables:

- Canonical `VoiceChannel` routing table for `chief_of_staff`, `hey_val`, `alert`, and `field_worker`.
- Kokoro implementation verified as the default local synthesizer.
- Provider registration mechanism for Piper, XTTS-v2, and cloud providers.
- Policy metadata per provider: latency, privacy tier, cloning support, cost class.

Implementation order:

1. Verify Kokoro path end to end.
2. Add Piper as the low-latency streaming option.
3. Reserve XTTS-v2 behind a feature flag for later persona work.
4. Add ElevenLabs or Cartesia only through explicit policy checks.

Acceptance criteria:

- A caller can request a voice channel without knowing which engine renders it.
- Provider selection can be made by policy instead of hardcoded call sites.

### 3. val-switch orchestration

Objective: make val-switch the single orchestration layer for omni-channel voice behavior.

Deliverables:

- Voice event schema for inbound utterances, transcripts, call state, and TTS actions.
- Conversation-thread linkage between voice, email, Teams, and WhatsApp.
- Policy engine hooks for business hours, retries, escalation, privacy tier, and budget.
- TTS channel selection rules driven by context.

Key responsibilities to implement:

- Route outbound call requests to the right carrier or channel.
- Stitch transcripts into the existing conversation entity.
- Enforce privacy and cost rules before cloud STT or TTS use.
- Insert voice security checks where sensitive calls require them.

Acceptance criteria:

- A voice interaction appears as part of the same conversation history as the user's other channels.
- val-switch can decide whether a request stays local or is allowed to use cloud voice services.

### 4. Telephony wire stack

Objective: stand up the locked Ring 0 stack for PSTN and call control.

Deliverables:

- Kamailio container for SIP edge and fraud-control basics.
- FreeSWITCH container for media handling.
- drachtio-server plus a Node SIP app for call control in the current local implementation.
- MeshCentral container for remote device operations and SOP capture.
- Sidecar adapters so each service appears as a mesh peer.

Integration sequence:

1. Bring up Kamailio and FreeSWITCH with internal connectivity.
2. Layer in drachtio call control for the local MVP, keeping the webhook/control-plane seam explicit for future evolution.
3. Add faster_whisper as the default telephony STT sidecar.
4. Connect sidecars to mesh RPC surfaces.
5. Validate a full PSTN inbound and outbound test flow.

Acceptance criteria:

- A phone call can enter through Kamailio, be handled by the SIP control layer, reach FreeSWITCH media, be transcribed by the STT sidecar, and then be forwarded into val-switch.

### 5. Telephony STT sidecar

Objective: provide transcription only for call media that cannot reach the device model directly.

Default choice:

- `faster_whisper`

Fallback matrix:

- `whisper.cpp` for no-GPU and low-memory hosts
- `WhisperLive` when partial transcripts and barge-in detection matter
- cloud STT only when policy allows it

Deliverables:

- Audio fork from FreeSWITCH into the sidecar.
- Transcript event contract for val-switch ingestion.
- Configurable sidecar backend selection.
- Privacy-tier enforcement around cloud fallback.

Acceptance criteria:

- Telephony transcripts arrive in val-switch with enough timing and call metadata to support routing and threading.

## Recommended phased rollout

### Phase 1: Foundations

- Clean up and consolidate the duplicated voice substrate spec.
- Identify the target repos for `val-desk`, `val-switch`, `val-core`, and deployment infrastructure.
- Define the shared event and schema contracts before implementation starts.

### Phase 2: Device MVP

- Ship the laptop microphone -> Gemma 4 E4B -> transcript/intent/response loop.
- Verify Kokoro playback through existing voice channels.
- Capture latency, CPU, memory, and quality metrics.

### Phase 3: TTS expansion

- Formalize provider registration behind `VoiceChannel`.
- Add Piper for interruptible and low-latency use cases.
- Defer XTTS-v2 and cloud TTS behind flags.

### Phase 4: Telephony infrastructure

- Deploy Kamailio, FreeSWITCH, the selected SIP control layer, and later MeshCentral.
- Add mesh sidecars and health checks.
- Validate internal call flows before external carrier onboarding.

### Phase 5: Telephony transcription and orchestration

- Add faster_whisper sidecar and transcript ingestion.
- Thread telephony transcripts into val-switch conversations.
- Enforce routing, privacy, escalation, and budget policies.

### Phase 6: Production hardening

- Add observability, recordings policy, retries, and failure handling.
- Add cloud STT/TTS policy gates where allowed.
- Run end-to-end tests for device voice, PSTN voice, and cross-channel threading.

## Cross-cutting requirements

### Contracts

- Shared schema for transcript, intent, response, call state, and TTS request events.
- Stable correlation IDs across device, telephony, and conversation systems.

### Observability

- Structured logs for audio capture, model inference, TTS synthesis, call control, and sidecar transcription.
- Metrics for latency, error rate, inference duration, and cloud-policy bypass attempts.

### Security and privacy

- Explicit privacy tiers per interaction.
- Cost and provider allowlists enforced in val-switch.
- Voice biometric and liveness checks added only at sensitive call boundaries.

### Operations

- Health checks for each Ring 0 component and sidecar.
- Clear fallback behavior when local inference or telephony STT fails.
- Deployment profiles for laptop, HQ host, and future GPU VM.
